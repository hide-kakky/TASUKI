# 16_デバッグの基本 🐛

「動かない！」と叫ぶ前にできることがあります。
プログラミング時間の半分はデバッグです。デバッグが上手くなれば、開発スピードは倍になります。

## 1. エラーメッセージを読む (Read the error)
英語の赤い文字に怯えないでください。そこに答えが書いてあります。

- `Null check operator used on a null value`: nullじゃないと思って `!` を使った変数が null だった。
- `RangeError (index): Invalid value`: 配列の範囲外（3個しかないのに4番目を取ろうとした）にアクセスした。
- `RenderFlex overflowed`: 画面からはみ出してる（レイアウトエラー）。

**最初の一行**だけでなく、**Stack Trace**（どのファイルの何行目で起きたか）を見ましょう。自分の書いたファイル名 (`package:tasuki_app/...`) を探すのがコツです。

## 2. 分割統治法 (Divide and Conquer)
どこが悪いかわからない時は、問題を切り分けます。

- **APIは動いてる？**: Postman や curl で叩いてみる。
  - OKならアプリ側の問題、NGならサーバー側の問題。
- **データは来てる？**: `print()` やブレークポイントで変数の中身を見る。
- **この行までは動いてる？**: 処理の途中々にログを仕込む。

## 3. ラバーダッキング (Rubber Ducking)
机の上のアヒルのおもちゃ（ラバーダック）に、コードの説明をする手法です。
「まずここで変数を初期化して、次にループで... あ！」
**人に説明しようとして整理しているうちに、自分で間違いに気づく** という現象には名前がついています。

## 4. 最小再現コード (MCVE)
複雑なアプリのまま悩むより、バグを再現させるための「最小のコード」を書いてみましょう。
意外と「余計な設定」が原因だったりします。質問するときも、これがあると回答率が爆上がりします。

## 📚 用語集 (Glossary)

| 用語 | 意味 | 補足 |
|------|------|-----|
| **Bug (バグ)** | 不具合 | 由来は本物の虫 |
| **Stack Trace** | エラー発生時の足跡 | 関数がどう呼ばれてきたかの履歴 |
| **Breakpoint** | 一時停止ポイント | デバッガーで処理を止める場所 |
| **Log (ログ)** | 記録 | `print` や `console.log` |
| **Reproducible** | 再現性がある | 誰がやっても同じ条件でバグる |
| **Flaky** | 不安定 | たまにバグる、タチの悪いやつ |
