# 12. 実装の流れ

## 🎯 このレッスンで学ぶこと

- アプリ開発の全体フロー
- 設計・実装・テスト
- Gitを使った開発
- TASUKIでの実践

---

## 1. 開発の4ステップ

プログラミングはいきなりコードを書きません。

```
1. 設計（Plan）: 何を作るか決める
   ↓
2. 実装（Code）: コードを書く
   ↓
3. テスト（Test）: 正しく動くか確認する
   ↓
4. デプロイ（Deploy）: 本番で使えるようにする
```

---

## 2. 設計（Plan）

**「何を作るか」を明確にするフェーズ**

TASUKIでは、`docs/` フォルダのドキュメントがこれにあたります。

- **要件定義**: 誰が何のために使う？（`TASUKI_requirements.md`）
- **データベース設計**: どんなデータを保存する？（`TASUKI_database_schema.md`）
- **画面設計**: どんな画面が必要？

**初心者のコツ**:
いきなり書き始めず、紙やメモ帳に「やりたいこと」を書き出してみましょう。

---

## 3. 実装（Code）

**実際にコードを書くフェーズ**

### TASUKIでの進め方

1. **マイグレーション**: データベースのテーブルを作る
   - `supabase/migrations/` にSQLファイルを作る

2. **バックエンド**: サーバー側の処理を作る
   - Edge Functions を実装する

3. **フロントエンド**: 画面を作る
   - FlutterでUIとロジックを作る

**なぜこの順番？**
データ（DB）がないとサーバー処理が書けず、サーバー処理がないと画面が動かないからです。
**「下流から上流へ」** 作るのが基本です。

---

## 4. テスト（Test）

**バグがないか確認するフェーズ**

- **単体テスト**: 関数ひとつひとつが正しく動くか？
- **結合テスト**: 画面とサーバーがつながっているか？
- **E2Eテスト**: ユーザーの操作通りに動くか？

TASUKIには `docs/TASUKI_e2e_scenarios.md` というテスト手順書があります。

---

## 5. Gitを使った開発フロー

Git（ギット）は、ゲームの「セーブポイント」を作るツールです。

```
1. ブランチを切る（新しい作業場所を作る）
   git checkout -b feature/add-login

2. コードを書く

3. コミットする（セーブする）
   git add .
   git commit -m "ログイン画面を作成"

4. プッシュする（クラウドに保存）
   git push origin feature/add-login

5. プルリクエスト（マージ依頼）
   「これを取り込んでください！」と依頼する
```

---

## 6. TASUKIコードサンプル集の使い方

`docs/TASUKI_code_samples.md` には、すぐに使えるコードが載っています。

**使い方**:
1. 実装したい機能を探す（例：Edge Functions）
2. コードをコピーする
3. 自分のファイルに貼り付ける
4. 必要なら少し修正する

これを**写経（しゃきょう）**と言います。
最初はコピペでも、**「なぜ動くのか？」**を考えながら読むことで力がつきます。

---

## 📝 まとめ

- 開発は 設計 → 実装 → テスト → デプロイ の順
- 実装は DB → バックエンド → フロントエンド の順
- Gitでこまめにセーブ（コミット）する
- サンプルコードを活用して学ぶ
