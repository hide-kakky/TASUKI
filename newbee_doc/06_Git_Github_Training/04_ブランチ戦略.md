# 04_ブランチ戦略 🌿

チームで開発する際、適当にブランチを作るとカオスになります。
代表的な運用ルール（ブランチ戦略）を知っておきましょう。

## 1. GitHub Flow
シンプルでモダン。TASUKIのような継続的デプロイメントに向いています。

1. **main**: 常にデプロイ可能な正解の状態。直接コミット禁止。
2. **feature/xxx**: 新機能や修正用。mainから生やし、mainに戻す(PR)。

### ルール
- main ブランチは常に守る（CIテストを通ったものしか入れない）。
- 作業が終わったらすぐに Pull Request (PR) を出す。
- マージされたら作業ブランチは消す。

## 2. Git Flow (参考)
もっと複雑で厳格なフロー。
`develop`（開発用）, `release`（リリース準備）, `master`（本番用）などを使い分けます。
大規模でリリースサイクルが長い（数ヶ月に1回など）プロジェクト向けです。今はあまり採用されません。

## 3. ブランチの命名規則
誰が見ても中身がわかる名前をつけましょう。

- **feature/**: 新機能 (例: `feature/login-screen`)
- **fix/**: バグ修正 (例: `fix/camera-crash`)
- **refactor/**: 挙動を変えずにコード整理 (例: `refactor/folder-structure`)
- **docs/**: ドキュメントのみ (例: `docs/update-readme`)

## 4. コンフリクトの予防
「マージしようとしたらコンフリクトした！」を防ぐコツ。

- **こまめに `git pull origin main` する**: 自分の作業中にも、mainは他の人の手で進んでいます。最新を取り込みながら作業しましょう。
- **PRを小さくする**: 変更範囲が小さければ、ぶつかる確率も減ります。

| **Branch Strategy** | ブランチ戦略 | チームのGit運用のルール |
| **Feature Branch** | 機能ブランチ | 作業用の短期ブランチ |
| **Main/Master** | メインブランチ | 幹となる一番大事なブランチ |
| **Production Ready** | 本番OK | いつでもリリースできる品質であること |
| **Prefix** | 接頭辞 | `feature/` などの頭につける文字 |
