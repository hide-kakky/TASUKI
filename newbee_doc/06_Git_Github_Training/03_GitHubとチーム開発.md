# 03. GitHubとチーム開発 🤝

複数人で開発する場合、全員が同時に `main` ブランチ（本番用の主要な履歴）を変更してしまうと、変更が衝突して壊れてしまいます。
それを防ぐためのルールと機能について学びます。

## GitHub Flow (開発の流れ)

1. **ブランチを切る (Branch)**
   - `main` から枝分かれした作業用ブランチ（例: `feature/new-login-ui`）を作ります。
   - `main` は常にきれいに保ち、作業はすべてこの専用ブランチで行います。

2. **コミット & プッシュ**
   - 作業用ブランチで開発を進め、GitHubにプッシュします。

3. **プルリクエスト (Pull Request / PR)**
   - GitHub上で、「私の変更を `main` に取り込んでください！」と依頼を出します。
   - これがコードレビューの場になります。

4. **レビュー & マージ**
   - チームメンバーがコードを確認し、OKなら `main` に結合（マージ）します。

## コマンドでの操作

### ブランチの作成と切り替え
```bash
# 新しいブランチを作って切り替える
git checkout -b feature/新しい機能名
```

### コンフリクト (衝突) とは？
AさんとBさんが「同じファイルの同じ行」を同時に別の内容に書き換えてしまった場合、Gitはどちらを採用していいかわからないため、**コンフリクト**が発生します。

#### 対処法
1. エディタで衝突しているファイルを開く。
2. `<<<<<<<` `=======` `>>>>>>>` という記号で囲まれた部分を確認し、正しい内容手動で書き直す。
3. 再度 `git add` → `git commit` する。

## チーム開発の心得
- **こまめに `git pull`**: 作業を始める前には必ず最新の状態を取り込みましょう。
- **PRは小さく**: 何日もの作業をまとめて1つのPRにすると、レビューが大変になります。機能単位で細かく出しましょう。
