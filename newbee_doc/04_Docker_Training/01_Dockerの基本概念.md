# 01. Dockerの基本概念 🐳

## 1. Docker（ドッカー）とは？
一言で言うと、**「どこでも同じように動く、隔離されたアプリケーション実行環境（コンテナ）を作るツール」** です。

### 🍔 「お弁当箱」で例えると...
あなたは「特製カレー」を作って友人に振る舞いたいとします。

- **Dockerを使わない場合（直接インストール）**:
  - 友人の家のキッチン（PC環境）に行き、そこで調理します。
  - 「あの調味料がない！」「コンロの火力が違う！」といったトラブル（環境差異）で、いつもの味が再現できないかもしれません。

- **Dockerを使う場合（コンテナ）**:
  - あなたは自宅のキッチンごと「お弁当箱（コンテナ）」に詰め込みます。
  - 友人の家では、そのお弁当箱を開けるだけ。
  - **どこで開けても、中身はあなたの作った環境そのもの** なので、確実に「いつもの味」が再現できます。

## 2. なぜDockerが必要なのか？

### 「私の環境では動くんですけど...」問題の解決
開発現場で最も恐ろしいこの言葉。
AさんのPCでは動くのに、BさんのPCではエラーになる。本番サーバーでは動かない。
これは、OSのバージョン、インストールされているライブラリ、設定ファイルなどの微妙な違いが原因です。

Dockerを使えば、OSからライブラリ、アプリまでをひとまとめ（イメージ）にして配布できるため、**「誰がどこで動かしても同じ結果になる」** ことが保証されます。

## 3. 重要な2つのキーワード

Dockerを理解するには、以下の2つの概念の区別が最重要です。

### 📜 イメージ (Image) = 「レシピ兼・金型」
- アプリケーションを実行するために必要なすべて（コード、ランタイム、ライブラリ、設定）が記録された**読み取り専用のテンプレート**。
- 例：「iOSのインストールディスク」「RPGゲームのカートリッジ」「鯛焼きの金型」。
- 実体はファイルなので、動いてはいません。

### 📦 コンテナ (Container) = 「実体化した料理」
- イメージをもとに作成され、実際にメモリ上で動いている**実行プロセス**。
- 例：「起動中のiPhone」「プレイ中のゲーム」「焼き上がった鯛焼き」。
- 1つのイメージ（金型）から、複数のコンテナ（鯛焼き）を量産できます。
- コンテナを削除しても、元のイメージ（金型）は消えません。

## 4. 仮想マシン (VM) との違い
よく比較されますが、Dockerは「軽い」のが特徴です。

- **仮想マシン (VM)**:
  - PCの中にもう一台PCを作るようなもの。OS（WindowsやLinux）を丸ごと起動するため、起動に数分かかり、メモリも数GB消費します。
- **コンテナ (Docker)**:
  - OSの核（カーネル）はホストPCのものを共有し、アプリに必要な部分だけを隔離します。
  - 起動は数秒、メモリ消費も最小限です。

## 5. TASUKIプロジェクトでの利用
このプロジェクトでは、バックエンドの **Supabase** をローカル環境で再現するためにDockerを使用しています。
`supabase start` というコマンドを打つと、裏側では「データベース」「認証サーバー」「APIサーバー」という複数のコンテナが連携して立ち上がっています。

次章からは、実際にこのコンテナを自分の手で操作してみましょう。
