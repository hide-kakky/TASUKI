# 15_良いIssueの書き方 📝

Issue（チケット）は仕事の「依頼書」です。
ここが曖昧だと、間違ったものが出来上がったり、何度も確認が発生してしまいます。

## 1. 良いIssueの基本テンプレート

```markdown
## 概要 (Summary)
ビデオ録画画面で、録画ボタンが反応しないときがある。

## 再現手順 (Steps to Reproduce)
1. アプリを起動
2. タイムライン画面からカメラボタンを押す
3. 録画ボタンを0.1秒だけタップする

## 期待する動作 (Expected Behavior)
「長押ししてください」というヒントが表示される、もしくは短すぎてキャンセルになるべき。

## 実際の動作 (Actual Behavior)
何も起きず、ボタンが無反応になる（フリーズしたように見える）。

## 環境 (Environment)
- iPhone 14 Pro
- iOS 17.2
- アプリバージョン 1.0.3
```

## 2. 3つのポイント

### A. 事実と推測を分ける
- 事実: 「ボタンを押しても画面が変わらない」
- 推測: 「たぶんカメラの初期化に失敗していると思う」
混ざるとミスリードになります。「推測ですが〜」と明記しましょう。

### B. スクショ・画面録画最強説
「右上のボタンが〜」と書くより、そのボタンを赤枠で囲った画像を1枚貼るほうが100倍伝わります。

### C. DoD (Definition of Done) を明確に
「使いやすくする」のような抽象的なゴールはNGです。
「タップ領域を44px以上に拡大する」のように、誰が見ても「完了した」と判定できる条件を書きましょう。

## 3. タスク分解
大きな機能（例: 「ログイン機能」）は、そのままIssueにすると大きすぎます。
- 画面レイアウト作成
- バリデーション実装
- API連携
- テストコード作成
のように、1日〜3日で終わるサイズに分割（Sub-task）しましょう。

## 📚 用語集 (Glossary)

| 用語 | 意味 | 補足 |
|------|------|-----|
| **Issue (イシュー)** | 課題、タスク、バグ報告 | 仕事の単位 |
| **Assignee (アサイン)** | 担当者 | 誰がやるか |
| **Label** | 分類タグ | `bug`, `enhancement`, `wontfix` など |
| **Milestone** | 期限・目標地点 | 「Ver 1.0 リリース」など |
| **DoD** | Definition of Done | 完了の定義。ゴール条件 |
| **Reproduce** | 再現する | バグを自分の環境でも起こすこと |
