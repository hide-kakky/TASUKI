全体像・学習ルート・役割別のおすすめを1ページでまとめました。迷ったらここに立ち返ってください。

---

## 1. まず全体像をつかむ
- TASUKIは「動画×AI×コミュニティ」のモバイルアプリ。FlutterでUIを作り、Supabaseでバックエンドをサーバーレスに構築。Muxで動画配信し、Geminiで解析を行います。
- データの流れ: モバイルUI → Supabase REST/RPC → DB(PostgreSQL) → AI処理はEdge Functions経由でGemini → Muxでストリーミング配信。
- セキュリティの軸: Supabase Authで認証、RLSで認可、Muxの署名URLで動画保護、CI/CDで自動テストを回す。

## 6. 全体像マップ（プロダクトの流れ）
```
[ユーザー] 
   ↓ (操作)
[Flutterアプリ]  --HTTP-->  [Supabase (Auth/REST/RPC/DB)]
                               ↓ (Edge Functions) → [GeminiでAI解析]
                               ↓ (Storage/DB/RLS)
              ← HLS再生 ← [Mux でエンコード/配信] ← (動画アップロード)
```
- フロント: FlutterでUI、HLSで動画再生
- バックエンド: Supabase (Auth/DB/API/RLS)、Edge FunctionsでAIや署名URL発行
- 動画: Muxがアップロード→エンコード→HLS配信
- セキュリティ: Authで認証、RLSで認可、Muxの署名URLで動画保護

## 5. よくある詰まりポイントとヒント
- **認証/認可の違い**: ログインは「あなた誰？」(Auth)、RLSは「このデータ見ていい？」(Authz)。  
- **非同期処理**: Flutterの`Future`/`Stream`とSupabaseのリアルタイムを混同しない。UIは`StreamBuilder`で分離。  
- **動画配信**: アップロード→Mux→HLS再生の3段階。トークン付きURLで保護する。  
- **CI/CD**: テストの粒度を小さく。Lint/Unit/UI/E2Eの順で壊れた箇所を早期に検知。  
- **Git運用**: 1PRを小さく、1テーマに絞る。コミットメッセージは「何をなぜ」を短く。
